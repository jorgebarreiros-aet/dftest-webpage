1 — What DFTest does

DFTest is designed to support graph-based dataflow testing of Java source code. Given a Java method, it can:

  • Derive the Control Flow Graph (CFG)
  • Identify all DU-pairs and DU-paths
  • Determine the test paths traversed by a provided test file
  • Highlight which requirements are satisfied under the all-uses and all-DU-paths coverage criteria
  • Produce an HTML coverage report across multiple methods

Because it builds and displays the CFG interactively, DFTest can also assist in designing and evaluating tests based on structural graph coverage, even without a test file.

2 — Exploring the interface

Step 1

Launch DFTest

Download dftest.jar and run it from a terminal: 

java -jar dftest.jar

The GUI will open showing a default Java method already loaded and analysed.

[ Screenshot: initial DFTest window ]

The window is divided into three main areas:

  • Top-left — Java source code editor
  • Right — Control flow graph
  • Bottom — DU-Pairs and DU-Paths panels, with a variable filter

3 — Reading the CFG

Each node in the graph corresponds to a statement or group of statements in the source code. Nodes display the variables they define (def) and use. Edges carry the condition variable uses for decision nodes.

Step 2

Click a node to highlight source code

Click on any node in the graph — for example, the node corresponding to int s = 0;. The matching line in the source editor will be highlighted, making the relationship between graph and code explicit.

[ Screenshot: node selected, source line highlighted ]

At the bottom of the screen you will find the DU-Pairs and DU-Paths computed for the current method.

[ Screenshot: DU-Pairs and DU-Paths panels ]

Each entry follows a straightforward notation:

DU-Pair format

(6, edge(1,3), n@1:21) — variable n (declared at line 1, col 21), defined at node 6, used at edge (1→3)

(5, 3, s@2:9) — variable s (declared at line 2, col 9), defined at node 5, used at node 3

DU-Path format

DU-path(n@1:21): 6 → 5 → 4 → 1 → 3 [edge (1,3)] — a def-clear path from node 6 to the use of n at edge (1,3)

4 — Merging linear nodes

Sequences of nodes with no branching can be merged into a single basic block. This simplifies the graph and the DU-pair notation without losing information.

Step 3

Enable node merging

Click the Merge nodes toggle in the toolbar. Nodes that form an uninterrupted linear sequence will be combined. Their merged name (e.g. [6,5,4]) reflects all original node ids.

[ Screenshot: graph after merging nodes ]
Tip: After merging, clicking a merged node highlights all the corresponding source lines. DU-pair and path names update automatically to use the merged node identifiers.

5 — Entering and checking test paths

Test paths are entered in the Test paths area below the source editor. Each line is one path — a space-separated sequence of node ids ending in exit.

Step 4

Type a test path and check coverage

Enter a test path, for example:

6,5,4 1 exit

You may use either the merged node name 6,5,4 or the individual ids 6 5 4 — both are accepted. Then click Check coverage.

[ Screenshot: test path entered, coverage checked ]

Each DU-pair and DU-path is now prefixed with a coverage mark:

  • ✓ — requirement satisfied by the test suite
  • ✗ — requirement not yet satisfied
  • ● — satisfied specifically by the currently selected path

The status bar at the bottom shows a summary, e.g. 7/10 DU-Pairs (all-uses) | 7/10 DU-Paths (all-DU-paths).

Filtering: Use the Filter by variable dropdown above the panels to focus on the DU-pairs and paths for a single variable. This is especially useful for methods with many variables.

6 — Clicking paths and DU entries

Step 5

Visualise a test path on the graph

Click on any line in the Test paths area. The corresponding nodes will be highlighted in the graph, giving a visual confirmation of the path you entered.

[ Screenshot: test path highlighted on graph ]
Step 6

Import a DU-path as a test path

Click on any entry in the DU-Paths panel. The node sequence of that path is automatically added to the Test paths area, ready to be completed and checked. For example, clicking an uncovered path such as: 

DU-path(n@1:21): 6,5,4 → 1 → 3,2 [edge (1,3)]

will insert 6,5,4 1 3,2 into the test path area. You can then extend it to reach exit if desired.

[ Screenshot: DU-path clicked, added to test paths ]

7 — Generating a coverage report

Step 7

Export an HTML report

Once you have test paths for one or more methods, click the 📊 Report button on the right of the toolbar. A save dialog will appear; choose a location and the report will open automatically in your browser.

The report includes, for each method: a coverage summary, the test paths used, and the full annotated list of DU-pairs and DU-paths with ✓/✗ markers. An overall summary across all methods is shown at the top.

[ Screenshot: HTML coverage report ]

8 — Saving and loading a project

Closing DFTest without saving will discard all test paths and coverage results. Use the project system to preserve your work.

Step 8

Save a session

Click 💾 Save project in the toolbar. A .cfgproject file will be saved containing the source code, test paths, coverage results, and configuration. Enable Embed source in the save dialog if you want the source to travel with the project file rather than referencing an external path.

To resume, click 📂 Load project and select the file. After loading, press Check coverage once to restore the coverage annotations.

9 — Working with Java files

Step 9

Open a .java file

Click Open file… and select any .java source file. If the class contains multiple methods, use the method selector dropdown in the toolbar to switch between them — each has its own CFG, DU-pairs, and test paths.

[ Screenshot: method selector dropdown with multiple methods ]
Tip: For methods with many nodes, switch to the Spring layout in Settings for a cleaner graph. DFTest selects this automatically for larger methods.
Note: If you edit the source code directly in the editor, the ▶ Analyze button will turn red, indicating the CFG needs to be rebuilt. Click it to reanalyse — existing test paths will be cleared.

10 — Importing traces from a test file

Instead of entering test paths manually, DFTest can run your test code, collect execution traces, and populate the test path area automatically.

Step 10

Set a test file and run

Click 📄 Set test file… in the toolbar. In the dialog, browse to a .java file containing test methods. Tests are identified by:

  • Methods annotated with @Test (JUnit 4/5)
  • Methods whose name starts with test

If your test file uses JUnit, click Add JUnit 5 (or Add JUnit 4) in the dialog — DFTest will search your local Maven repository and add the required jars automatically. If automatic discovery fails, add the jars manually via Add jar…, or remove the JUnit imports and annotations from the test file (the test* naming convention will still be recognised).

[ Screenshot: Set test file dialog with classpath entries ]
Step 11

Run the tests

Click ▶ Run tests in the toolbar. DFTest compiles the instrumented source, executes each test method, and adds the resulting execution traces to the Test paths area. Each path is annotated with the test method name as a comment, e.g.: 

20 19 18 17 16 15 14 2 4 13 12 11 5 7 6 5 3 2 1 exit  // testPatternFound

Press Check coverage to evaluate the collected traces against the DU criteria.

[ Screenshot: test traces loaded into test path area ]
Tip: Save the project after setting the test file so you do not have to configure the classpath again on the next session.

Exercises

Practice tasks

  1. Using the default method loaded on startup, achieve 100% all-uses coverage by adding test paths manually. Identify any redundant paths in your test suite and remove them.
  2. Open a Java file of your choice with at least one non-trivial method. Derive its CFG, inspect the DU-pairs, and design a minimal set of test paths that satisfies the all-uses criterion.
  3. For the same method, extend your test suite to achieve all-DU-paths coverage. How many additional paths were needed?
  4. Set a JUnit test file for the method above. Run the tests and check what percentage of DU-pairs and DU-paths the existing tests already cover. Use the coverage results to identify gaps and suggest new test cases.
  5. Generate an HTML report for all methods with test paths and review it. Export it and include it in your lab submission.
